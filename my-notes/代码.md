**键盘快捷键恢复最近关闭的标签页**：

| 操作系统 | 快捷键 |

适用于：Chrome、Edge、Firefox、Safari（Mac）等所有主流浏览器。

### Q：`--dangerously-skip-permissions` 有没有更简便的写法？

**A：三种方案，按场景选**：

| 场景 | 推荐方案 | 写法 |
|------|---------|------|
| 永久生效 | 环境变量 | `CLAUDE_CODE_PERMISSIONS=allow claude ...` 加入 `~/.zshrc` |
| 只省编辑确认 | flag | `claude --approve-edits` |
| 项目级配置 | 配置文件 | 项目根目录 `.claude.json` 设置 `"permissions": "allow"` |

### Q：`"allow": [...]` 和 `"allow": true` 有何区别？

**A：粒度不同**：

| 写法 | 效果 |
|------|------|
| `"allow": ["Bash(pip3:*)", "Read(/path/**)"]` | **精确白名单**，只放行匹配 pattern 的调用 |
| `"allow": true` 或 `"bypassPermissions": true` | **彻底放行**，不过任何权限校验 |

> ⚠️ `"dangerouslySkipPermissions": true` 等效于 `claude --dangerously-skip-permissions`，是最暴力的方式。

**结论**：如果 `settings.json` 已设置 `"dangerouslySkipPermissions": true`，则 `--dangerously-skip-permissions` flag 重复了，没必要加。

### Q：`~/.zshrc` 里 `export XXX="XXX"` 中间能加空格吗？

**A：不能，必须紧挨着。**

Shell 变量赋值**不能有空格**，正确写法：

```bash
export XXX="XXX"     # ✅ 正确
export XXX = "XXX"   # ❌ 错误
export XXX= "XXX"    # ❌ 错误
export XXX ="XXX"    # ❌ 错误
```

> ⚠️ `export VAR=value` 中 `=` 左右也不能加空格，等号两侧必须紧邻。

## 2026-04-13

### Q：`npx stdio-to-ws "npx @zed-industries/claude-code-acp" --port 3017` 这条命令是做什么的？

**A：把 Claude Code ACP 代理的 stdio 桥接成 WebSocket 服务（端口 3017），供 marimo 等 WebSocket 客户端连接。**

### 架构拆解

| 组件 | 作用 |
|------|------|
| `npx stdio-to-ws` | 协议转换层：把 stdio 转成 WebSocket |
| `npx @zed-industries/claude-code-acp` | ACP 适配器：Claude Code SDK → ACP 协议 |
| `--port 3017` | 监听端口，marimo 通过 `ws://localhost:3017` 连接 |

### 数据流

```
marimo (WebSocket 客户端)
    ↕ ws://localhost:3017
stdio-to-ws（协议转换层）
    ↕ stdio
@zed-industries/claude-code-acp
    ↕ Claude Code SDK
Claude Code（AI 能力）
```

### 两个端口的区别

| 服务 | 端口 | 说明 |
|------|------|------|
| marimo 自身 | 默认 2718 | 你打开 marimo notebook 的地址 |
| Claude Code ACP 代理 | **3017** | marimo 连接 AI Agent 的 WebSocket 地址 |

> ⚠️ 这是**两个独立进程**：marimo 跑在 2718，`stdio-to-ws` 跑在 3017，端口不同、互不冲突。

### Q：marimo 设置面板里的 optional features 各有什么作用？

**A：共 9 个可选功能，分 4 类**：

### 数据与 SQL 类

| 功能 | 包 | 作用 |
|------|-----|------|
| SQL Cells | `duckdb`, `sqlglot` |  notebook 内直接写 SQL，duckdb 高速进程内执行，sqlglot 做语法解析验证 |
| Charts in Datasource Viewer | `altair` | datasource 面板浏览 DataFrame 时自动生成分布图表；不装则只显示表格 |
| Fast Server-Side Charts | `vegafusion`, `vl-convert-python` | 大图表在**服务端渲染**，避免浏览器假死 |

### AI 与语言类

| 功能 | 包 | 作用 |
|------|-----|------|
| AI Features | `openai` | 编辑器内置 AI 助手（自动补全/聊天/代码生成），可对接 OpenAI 或 Anthropic 等兼容 API |
| Connect to MCP Servers | `mcp` | 让 marimo 作为 MCP 客户端/服务器，扩展 AI 工具和数据源 |
| Language Server Protocol (LSP) | `python-lsp-server`, `websockets` | **IDE 级代码智能**：跳转定义、悬停文档、诊断、自动补全；需重启服务器激活 |

### 工具类

| 功能 | 包 | 作用 |
|------|-----|------|
| Formatting | `ruff` | 保存/运行 cell 时自动格式化代码，保持风格一致 |
| Export as IPYNB | `nbformat` | 导出为 Jupyter `.ipynb` 格式，便于分享给 JupyterLab/VS Code 用户 |
| Autorun Unit Tests | `pytest` | 编辑器内集成 pytest，notebook cell 中定义的测试自动运行并内联显示结果 |

> ⚠️ 所有功能均已安装，无需额外配置。

## 2026-04-21

### Q：Onyx 部署中，`docker compose down` 会删除用户数据吗？

**A：普通 `down` 不会，加 `-v` 才会**：

| 命令 | 数据安全？ |
|------|-----------|
| `docker compose down` | ✅ 安全 — 只删除容器和网络，**卷（volumes）保留** |
| `docker compose down -v` | ❌ **数据全删** — `-v` 会删除所有 named volumes |
| `docker compose rm` | ✅ 安全 — 只删除已停止的容器 |

### Q：本地开发会影响到 Docker 中的用户数据吗？

**A：不会 — 本地开发用不同端口，Docker 保持原样**：

```bash
# Docker 继续在 8080/3000 运行
# 本地开发用 8081/3001

cd backend
source .venv/bin/activate
AUTH_TYPE=basic uvicorn onyx.main:app --reload --port 8081

# 另一个终端：
cd web
PORT=3001 npm run dev
```

访问 **http://localhost:3001** — 调用本地 8081 API，再连 Docker 的 DB/Redis/MinIO。

### Q：本地测试完成后，如何把改动应用到 Docker 而不丢失用户数据？

**三种方案**：

| 方案 | 操作 | 数据安全？ |
|------|------|-----------|
| **A. docker cp** | `docker cp backend/onyx onyx-api_server-1:/app/onyx` 然后 `docker restart onyx-api_server-1` | ✅ |
| **B. 只重建 API 服务** | `cd deployment/docker_compose && docker compose up -d --build api_server` | ✅ |
| **C. Git 提交后 exec 拉取** | `docker exec -it onyx-api_server-1 bash` → `git pull` | ✅ |

**原理**：PostgreSQL 用户账号/登录信息存在 **Docker volume** 中，只要不 `down -v`，重建容器不会影响。

## 2026-04-22

### Q：Onyx 本地开发完成后部署到 Docker，如何保留已有数据？

**A：永远不要用 `-v`，重建用 `--build` 即可**：

```bash
# ✅ 正确：重建服务，保留 volume
docker compose up -d --build

# ❌ 错误：会删掉所有数据
docker compose down -v
docker compose rm -v
```

### 推荐工作流

```bash
# 只重建改过的服务
docker compose up -d --build api_server   # 后端改了
docker compose up -d --build web_server  # 前端改了

# 或者只是重启（最快）
docker compose restart api_server
docker compose restart web_server
```

### 验证 volume 还在

```bash
docker volume ls | grep onyx
```

### Q：如何让校园网的 Onyx (172.xx) 变成公网可访问？

**三种方案**：

| 方案 | 适用场景 | 操作复杂度 |
|------|---------|-----------|
| **Cloudflare Tunnel** | 快速验证，不需要服务器 | ⭐ 最简单 |
| **云服务器直接部署** | 长期使用 | 需购买服务器 |
| **frp 内网穿透** | 已有公网服务器 | 中等 |

### 方案一：Cloudflare Tunnel（5 分钟搞定）

```bash
cloudflared tunnel --url http://localhost:3003
```

得到 `*.trycloudflare.com` 域名，公开可访问。

### 方案二：云服务器部署

```bash
# 在服务器上
git clone onyx
cd deployment/docker_compose
docker compose up -d
```

域名 A 记录指向服务器公网 IP，用 80/443 端口。

### 方案三：frp 穿透

需要一台公网服务器作为中转，配置 frps/frpc。
