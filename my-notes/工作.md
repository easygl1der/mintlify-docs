## Mac Mini 外置 SSD 路径映射（Symlink）配置

**日期**：2026-04-06

---

### Q：如何让 ~/Projects 的访问落到外置 SSD 上？

**通过 symlink（符号链接）实现路径映射**：
```bash
mv ~/Projects ~/Projects_bak                              # 备份原目录
ln -s /Volumes/SSK\ SSD/Projects ~/Projects              # 创建 symlink
```
- **`~/Projects` → `/Volumes/SSK SSD/Projects`**
- 应用无感知，完全透明，读写删都落在 SSD 上

---

### Q：rm 删除 ~/Projects 下的文件会怎样？是否能在 Volumes 删除？

**两种路径删的是同一个真实文件**：

| 操作 | 删的是 |
|------|--------|
| `rm ~/Projects/foo.txt` | **SSD 上真实文件** |
| `rm /Volumes/SSK SSD/Projects/foo.txt` | **SSD 上真实文件** |

**没有区别**，symlink 对文件操作完全穿透。

> 但 `rm ~/Projects`（不带文件名）删的是 symlink 本身，真实文件不受影响。

---

### Q：什么操作不会穿透 symlink？

| 命令 | 行为 |
|------|------|
| `rm ~/Projects` | **删 symlink 本身**，真实文件不受影响 |
| `du -sh ~/Projects` | 只计算 symlink 大小（几字节） |
| `du -shL ~/Projects` | 计算真实文件大小 |
| `mv ~/Projects ~/foo` | 移动 symlink，真实目录不受影响 |

---

### Q：如何查看当前路径是逻辑路径还是真实路径？

```bash
pwd        # 逻辑路径（/Users/yueyh/Projects）
pwd -P    # 真实物理路径（/Volumes/SSK SSD/Projects）
```

---

### Q：什么操作是危险的高风险操作？

**直接对 SSD 真实目录操作**：
```bash
rm -rf /Volumes/SSK\ SSD/Projects/*   # ← 直接删 SSD 真实文件
```
**记住原则**：`~` 开头操作 symlink（相对安全）；`/Volumes/SSK SSD/` 开头操作真实文件（危险）。

### 修复 JSON API 403 错误 (2026-04-21 13:10)

**问题**：Onyx 连接 SearXNG 时返回 403，JSON 格式未启用

**原因**：`search.formats` 只配置了 `html`，缺少 `json`

**修复步骤**：
1. 编辑 `core-config/settings.yml`
```yaml

  formats:
    - html
    - json  # 添加这行
```
2. 重启容器
```bash
docker restart searxng-core
```

**验证**：
```bash

# 返回 JSON 结果 ✅
```

| inference_model_server | onyxdotapp/onyx-model-server | model_cache_huggingface | 模型缓存 |

### Volume 安全操作原则

| 命令 | 数据影响 |
|------|----------|
| `docker compose down` | ❌ **不删** volume，数据安全 |
| `docker compose down -v` | ⚠️ **删除** named volume，数据丢失 |
| `docker compose restart` | ✅ 只重启容器，volume 不变 |
| `docker compose up -d` | ✅ 重建容器，volume 保留 |
| `docker volume rm <name>` | ⚠️ 删除指定 volume，**不可逆** |
| `docker volume prune` | ⚠️ 删除所有未使用的 volume |

> **核心原则**：`down` 不加 `-v` = 安全；只有 `-v` 或显式 `rm` 才删数据。

### Docker Desktop 重启场景

**什么情况需要重启 Docker Desktop**：
- `docker compose` 命令全部超时/卡死
- `docker ps` / `docker info` 无响应
- 容器状态异常但无法通过 `docker compose` 修复

**重启步骤（数据零丢失）**：
```bash
# 1. 尝试杀掉卡住的 docker-compose 进程
pkill -9 docker-compose

# 2. 如果无效，重启 Docker Desktop
osascript -e 'quit app "Docker Desktop"'
sleep 8
open -a "Docker Desktop"

# 3. 等待 Docker Desktop 完全启动（右上角图标变绿）

# 4. 重新启动 onyx 容器
cd /Volumes/SSK\ SSD/Projects/onyx/deployment/docker_compose
docker compose up -d

# 5. 验证状态
docker compose ps
curl -s http://localhost:8080/health
```

## Onyx Docker 内存使用分析 (2026-04-23)

### Q：Docker 容器内存使用排名？

**Top 5 内存消耗：**

| 排名 | 容器 | 内存占用 | 占比 |
|------|------|---------|------|
| 1 | `onyx-index-1` (Vespa) | 2.823 GiB | 25.25% |
| 2 | `onyx-background-1` (Celery) | 2.563 GiB | 22.94% |
| 3 | `onyx-opensearch-1` | 1.001 GiB | 8.95% |
| 4 | `onyx-api_server-1` | 765.5 MiB | 6.69% |
| 5 | `deeptutor` | 282.4 MiB | 2.47% |

> Top 3 合计吃掉约 **57%** 内存

---

### Q：为什么这些容器用这么多内存？

**1. `onyx-index-1` (Vespa 向量数据库)**
- 存储文档向量嵌入用于语义搜索
- 缓存频繁访问的向量和文档片段
- 索引分片常驻内存以保证查询速度
- **这是预期最重的服务**

**2. `onyx-background-1` (Celery 异步任务)**
- 处理文档索引、嵌入生成、连接器同步
- 4+ 线程并发，缓冲文档内容和中间状态
- **内存随任务队列深度增长**

**3. `onyx-opensearch-1` (全文搜索引擎)**
- JVM 堆 + 原生内存用于索引缓冲
- 分片数据缓存加速查询

**4. `onyx-api_server-1` (FastAPI 服务)**
- 连接池、缓存的模型响应、会话数据

**降内存思路**：
- 降低 Celery 并发数
- 限制索引批处理大小
- 降低 Vespa 缓存层级
