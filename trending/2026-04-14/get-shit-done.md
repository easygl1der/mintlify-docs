## 基本信息
- **项目名称**: get-shit-done (GSD)
- **作者**: @gsd-build
- **描述**: 一个为 Claude Code 定制的轻量级且强大的元提示词（Meta-prompting）、上下文工程（Context Engineering）和规格驱动开发（Spec-driven Development）系统。
- **编程语言**: Markdown / Text (提示词驱动)
- **许可证**: MIT

## 项目简介
`get-shit-done` (GSD) 并非一个传统的软件库或二进制应用，而是一套**基于提示词的工程方法论系统 (Prompt Engineering Methodology System)**。它的核心目标是将 AI 编程从不可预测的“随缘编码 (Vibe Coding)”转变为一种可控、确定且高精度的工程实践。

GSD 认为，AI 编程的质量取决于输入信息的结构化程度。因此，它通过引入“规格驱动开发”和“元提示词治理”机制，引导 Claude 等 LLM 在生成代码前先进行严格的需求对齐和思考路径规划，从而将 AI 从简单的聊天助手升级为高精度的工程工具。

## 技术栈分析
该项目的技术栈极其特殊，它不运行在传统的操作系统运行时上，而是运行在 **LLM 的推理上下文 (Inference Context)** 中：

- **核心媒介**: **Markdown**。作为指令定义、知识传递和协议约定的唯一语言。
- **目标宿主**: **Claude Code** 及类似的 AI 编程助手。
- **核心技术方案**:
    - **元提示词 (Meta-Prompting)**: 构建高阶指令集，用于治理 LLM 的思考模式，定义其如何分析问题、如何生成代码以及如何进行自我校验。
    - **上下文工程 (Context Engineering)**: 通过对输入信息的精准组织（结构化），优化 LLM 的注意力机制 (Attention) 并提高关键信息的召回精度。
    - **规格驱动 (Spec-Driven)**: 借鉴传统软件工程的 SRS (软件需求规格说明书) 理念，为 AI 定义“刚性目标 (Rigid Target)”，消除模糊描述带来的随机性。

## 代码结构
由于项目不含可执行代码，其结构是按照**软件开发生命周期 (SDLC)** 映射而成的协议目录：

```text
get-shit-done/
├── prompts/    # 【执行引擎层】：存放元提示词。定义 AI 的行为准则、思考逻辑和代码生成标准。
├── specs/      # 【规格定义层】：提供技术需求和规格书的定义框架。确保在编码前完成精准的需求对齐。
├── templates/  # 【初始化层】：提供可复用的项目初始化模板，帮助用户快速搭建 GSD 工作环境。
├── examples/   # 【实践参考层】：通过真实案例展示“规格 $\rightarrow$ 提示词 $\rightarrow$ 代码”的完整链路。
└── README.md   # 【中央控制台】：定义 GSD 的核心哲学、工作流及使用指南。
```

## 依赖分析
- **依赖数量**: **零 (Zero)**。没有任何代码级依赖（无 `package.json` 或 `requirements.txt`）。
- **耦合度**:
    - **强耦合于 LLM 能力**: 系统的有效性完全依赖于底层模型（尤其是 Claude 系列）对复杂指令的理解和遵循能力。
    - **低耦合于开发环境**: 只要能够读取文本文件的环境（编辑器、IDE、AI 助手）均可无缝集成。
- **风险评估**: 唯一潜在风险为“模型漂移”。若 LLM 厂商更新模型导致对特定 Prompt 模式的响应发生变化，提示词可能需要重新调优。

## 可运行性评估
**评估结论：极高 (Very High) —— 协议执行维度**

- **运行方式**: 采用**“引导式加载”**。用户无需安装软件，仅需执行以下流程：
    `复制 templates` $\rightarrow$ `编写 specs` $\rightarrow$ `向 AI 注入 prompts` $\rightarrow$ `Claude 产出结果`。
- **部署成本**: 几乎为零，无需构建、编译或环境配置。
- **验证路径**: 用户可以通过 `examples/` 目录中的用例，快速验证该方法论在实际编程任务中对代码质量的提升效果。

## 技术亮点
1. **从“对话”到“协议”的升维**: GSD 不将 AI 视为聊天机器人，而将其视为一个需要“严格协议”驱动的编译器。通过 `specs` 强制 AI 在编码前进行思考，极大地降低了幻觉率。
2. **上下文熵值管理**: 通过结构化的上下文工程，在 LLM 有限的窗口内通过精准的信息组织降低信息熵，从而提升输出结果的确定性。
3. **元提示词的治理能力**: `prompts/` 目录中的指令不仅定义了“做什么”，更定义了 AI “如何思考”以及“如何自我校验”，实现了对 AI 行为的深度治理。

## 潜在问题
1. **认知门槛/学习曲线**: 要求用户具备软件工程基础（如理解 Specification），对于习惯于简单对话的初学者，编写严谨 Spec 的时间成本较高。
2. **Token 消耗增加**: 复杂的元提示词和详尽的规格书会占用较多输入 Token，在长项目或短上下文窗口的模型中，可能会导致有效信息被挤出。
3. **缺乏自动化校验**: 该方法论的正确性依赖于人工经验和 LLM 的主观执行，缺乏客观的、可量化的自动化测试手段。

## 总结与建议
`get-shit-done` 是一个将**软件工程理论转化为 Prompt 实践**的绝佳案例。它证明了在 AI 时代，“如何定义问题 (Spec)”的重要性已超过“如何写代码”。

该项目在 LLM 的不确定性之上构建了一层“确定性外壳”，将 AI 编程从一种“艺术”推向了“工程”。对于追求高精度、高可靠性 AI 辅助开发的团队而言，这套方法论具有极高的实用价值。

**建议：**
- **对于开发者**: 建议将 GSD 中的 `specs` 模板集成到团队的研发流程中，在向 AI 发出编码指令前，强制要求 AI 先生成并确认一份规格书。
- **优化方向**: 未来可尝试将这些提示词协议转化为可配置的 `.json` 或 `.yaml` 格式，以便与自动化 Agent 框架（如 LangGraph 或 AutoGPT）结合，实现规格驱动的自动化开发链路。
