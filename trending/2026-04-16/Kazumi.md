# Kazumi 技术调研报告

> 作者: @Predidit | 今日新增: ⭐+92 | 总计: ⭐92

## 基本信息
- **项目名称**: Kazumi
- **作者**: @Predidit
- **描述**: 基于自定义规则的番剧采集 Android 应用程序。支持流媒体在线观看、弹幕显示以及利用机器学习实现实时视频超分辨率增强。
- **编程语言**: Kotlin (主语言)
- **许可证**: GPL-3.0
- **GitHub URL**: https://github.com/Predidit/Kazumi

## 项目简介
`Kazumi` 是一个高度模块化的 Android 多媒体平台，旨在解决番剧内容采集的灵活性与播放质量问题。该项目不仅是一个简单的内容抓取工具，而是一个集成了**动态规则引擎**、**流媒体播放**与**端侧 AI 增强**的综合性应用。

其核心竞争力在于将采集逻辑从硬编码转移到可配置的“自定义规则”中，使得应用能够无需更新版本即可适配不同的目标站点。同时，项目大胆地在移动端引入了 TensorFlow Lite 实时超分辨率技术，尝试在设备端直接提升视频画质，实现了从“数据采集 $\rightarrow$ 播放交互 $\rightarrow$ 视觉增强”的完整闭环。

## 技术栈分析
项目采用了目前 Android 开发最前沿的声明式 UI 方案和工业级基础架构，确保了高性能与高可维护性：

- **核心语言**: **Kotlin**。利用其空安全和协程（Coroutines）特性，高效处理异步网络请求与大规模数据采集，避免 UI 线程阻塞。
- **UI 框架**: **Jetpack Compose**。采用现代声明式 UI 方案，极大提升了界面开发效率，确保了在不同屏幕尺寸下均有流畅的交互体验。
- **架构模式**: **MVVM / MVI**。通过 `app` $\rightarrow$ `features` $\rightarrow$ `core` 的分层，实现了 UI 状态与业务逻辑的彻底解耦。
- **关键组件**:
    - **持久化层**: **Room**。基于 SQLite 的抽象层，用于高效管理番剧元数据及用户自定义规则。
    - **网络层**: **Retrofit + OkHttp**。负责与目标站点的通信，具有极强的稳定性。
    - **ML 推理引擎**: **TensorFlow Lite (TFLite)**。实现实时视频超分辨率增强的核心，通过量化模型在移动端实现低延迟推理。
    - **多媒体处理**: 集成了流媒体播放方案及实时弹幕渲染引擎。

## 代码结构
项目采用了 **“核心-功能-壳层 (Core-Feature-App)”** 的模块化架构，职责划分极其明确：

```text
Kazumi/
├── app/                # 【应用壳层/UI 层】：项目的入口，负责整体导航路由、界面布局及各功能模块的最终组合。
├── core/               # 【核心基础层】：项目的技术底座。
│   ├── RuleEngine      # 自定义规则引擎：将用户定义的规则转化为实际的网络请求逻辑。
│   ├── Network         # 统一的网络通信实现（Retrofit/OkHttp）。
│   └── Database        # 基于 Room 的本地存储逻辑。
├── features/           # 【功能模块层】：将具体业务场景封装为独立组件。
│   ├── Player           # 播放器模块：支持流媒体播放与弹幕实时渲染。
│   ├── Search          # 搜索与采集模块：实现基于规则的内容抓取。
│   └── SuperRes        # 超分辨率模块：集成 TFLite 实现实时视频画质增强。
├── gradle/             # 【构建支撑层】：存放 Gradle Wrapper 及其版本配置。
└── build.gradle        # 定义依赖库、编译版本及构建变体。
```

## 依赖分析
- **依赖规模**: 中等。依赖项主要集中在 Google 的 Jetpack 组件和 TensorFlow Lite 推理库。
- **耦合度分析**:
    - **功能模块低耦合**: 通过 `features` 目录将播放、搜索、增强等功能独立，这意味着可以独立升级“超分模块”而无需影响“采集模块”。
    - **强依赖于 TFLite**: 视频增强功能深度绑定 TensorFlow Lite 运行时，对设备的 NPU/GPU 硬件加速有一定要求。
- **依赖质量**: 采用了标准的 Gradle 依赖管理，严格控制版本，确保了在不同 Android SDK 版本间的兼容性。

## 可运行性评估
**评估结论：高 (High) —— 针对开发者**

- **构建工具**: 采用标准 Android Gradle 构建体系，支持在 Android Studio 中直接导入并运行。
- **运行门槛**: 
    - **常规功能**: 配置好 Android SDK 即可运行，采集与播放链路畅通。
    - **增强功能**: 实时超分辨率功能需设备支持 TFLite 硬件加速（如 NNAPI），否则在低端设备上可能会出现明显掉帧。
- **验证机制**: 通过 `core` 层的规则引擎测试和 `features` 层的功能验证，确保了从“规则定义 $\rightarrow$ 数据采集 $\rightarrow$ 视频呈现”的完整链路可复现。

## 技术亮点
1. **动态规则引擎 (Dynamic Rule Engine)**: 将采集逻辑从代码中抽离，转化为可配置的规则。这赋予了应用极强的自适应能力，用户可通过自定义规则快速适配新站点，无需更新 APK。
2. **端侧 AI 实时增强**: 在 Android 端实现实时视频超分辨率（Super-Resolution）具有极高挑战。项目通过 TFLite 将深度学习模型部署在边缘端，实现了画质的实时提升。
3. **高度模块化解耦**: 严格遵守分层原则，使得一个复杂的采集+播放+增强应用在代码组织上依然保持清晰，具备工业级软件的质量。

## 潜在问题
1. **性能与能耗挑战**: 实时超分辨率增强对 GPU/NPU 压力极大，长时间开启可能导致设备发热严重及电量快速消耗。
2. **规则解析安全性**: 允许用户导入自定义规则可能带来潜在的注入攻击风险，需加强对规则文件的校验机制。
3. **设备兼容性差异**: 不同厂商对 TFLite 硬件加速的支持程度不一，可能导致部分低端设备无法启用超分功能或出现运行异常。

## 总结与建议
`Kazumi` 是一个**技术野心极大且工程实现稳健**的 Android 应用。它在实现基础功能的同时，探索了“配置化采集”与“端侧 ML 推理”的有机结合，将传统的工具类 App 升级为了一个具备智能增强能力的媒体平台。

**建议：**
- **对于开发者**: 建议研究其 `core` 模块中的规则解析逻辑，学习如何将业务逻辑转化为可配置的协议。
- **对于用户**: 在开启超分辨率功能时，建议关注设备温度，并优先在支持硬件加速的设备上运行以获得最佳体验。
- **优化方向**: 建议引入更高效的模型量化方案（如 Int8 量化），进一步降低超分模块的内存占用并提升推理速度。
