## 2. 项目简介

### 核心定位

marimo 是一个**响应式 Python 笔记本环境**，旨在解决传统 Jupyter Notebook 的核心痛点：

- **消除隐藏状态** — 响应式执行模型保证代码、输出、程序状态三者一致
- **Git 友好** — 笔记本存储为纯 `.py` 文件，而非 JSON 格式
- **可部署** — 支持脚本执行、Web 应用部署、WASM 浏览器运行
- **AI 原生** — 内置 AI 代码生成、数据感知补全

### 竞品对比

| 特性 | marimo | Jupyter | Pluto.jl | ObservableHQ |
|------|--------|---------|----------|-------------|
| 响应式执行 | ✅ | ❌ | ✅ | ✅ |
| 纯 Python 存储 | ✅ | ❌ (IPYNB) | ❌ (Julia) | ❌ |
| 内置 SQL 支持 | ✅ | ❌ | ❌ | ❌ |
| WASM 浏览器运行 | ✅ | ❌ | ❌ | ✅ |
| 内置包管理 | ✅ | ❌ | ❌ | ❌ |
| AI 原生编辑 | ✅ | ❌ | ❌ | ❌ |

### 灵感来源

项目明确致敬了 [Pluto.jl](https://github.com/fonsp/Pluto.jl)、[ObservableHQ](https://observablehq.com/tutorials)、以及 Bret Victor 的响应式编程理念。

### 4.2 核心模块详解

#### `_runtime/` — 响应式运行时引擎

这是 marimo 最核心的模块，负责实现响应式执行模型：

```
_runtime/
├── runtime/
│   ├── executor.py           # 单元执行器
│   ├── rerunner.py           # 依赖追踪与重运行
│   ├── defs.py               # 变量定义追踪
│   └── runner.py             # 主运行循环
├── dataflow.py               # 数据流分析
├── graph.py                  # 计算图构建
├── runtime_context.py        # 运行时上下文
└── suspension.py             # 延迟执行控制
```

**执行流程**:
1. 用户运行某个 Cell → AST 分析提取变量引用
2. 构建依赖图 (Dependency Graph)
3. 拓扑排序确定执行顺序
4. 仅执行受影响的下游 Cell
5. 删除 Cell 时自动清理其定义的变量 (消除隐藏状态)

#### `_server/` — ASGI Web 服务器

基于 Starlette 的 Web 服务器：

```

├── server.py                 # 主服务器
├── api/

│   ├── auth.py               # 认证
│   └── endpoints/            # 各端点
├── sessions/                 # 会话管理
└── file_router.py            # 文件路由
```

## 7. 技术亮点

### 亮点 1: 响应式执行模型

**问题**: Jupyter Notebook 的执行顺序依赖 Cell 物理位置，导致隐藏状态和难以复现的 bug。

**解决方案**: 基于 Python AST 的依赖图自动分析：

```python
# 用户代码
a = 1          # Cell 1
b = a + 1      # Cell 2 依赖 a
c = b * 2      # Cell 3 依赖 b

# 修改 Cell 1 后运行
a = 10         # 触发 → Cell 2 重运行 → Cell 3 重运行
```

**实现**: `_ast/` 模块解析 Python 代码为 AST，提取变量定义和引用，构建计算图。

### 亮点 2: Loro CRDT 实时协作

```json
// package.json 依赖
"loro-crdt": "^1.10.5",
"loro-codemirror": "^0.3.3",
```

marimo 使用 **Loro** (一个专为代码编辑优化的 CRDT 实现) 支持多人实时协作编辑：

- 无需 OT/Operational Transformation 的复杂性
- 自动冲突解决
- 支持离线编辑和后续同步
- 与 CodeMirror 6 深度集成

### 亮点 3: Narwhals 统一 DataFrame API

```python
# pyproject.toml
"`narwhals>=2.0.0`"

# 用户代码 — 自动兼容 Polars 和 Pandas
import narwhals as nw

df = nw.from_native(pandas_df)  # 或 polars_df
result = df.group_by("category").agg(nw.col("value").sum())
```

marimo 使用 **Narwhals** 提供统一的 DataFrame 操作接口，用户代码自动兼容 Polars 和 Pandas，无需关心底层实现。

### 亮点 4: 内置 SQL 支持

```python
import marimo as mo

# 定义 SQL 查询，依赖 Python 变量
result = mo.sql(
    f"""
    SELECT category, SUM(value) as total
    FROM df
    WHERE threshold > {threshold}
    GROUP BY category
    """
)
```

基于 **DuckDB** 的嵌入式 SQL 引擎，支持：
- 原生查询 Polars/Pandas DataFrame
- 查询 DuckDB 原生数据库
- 读写 CSV、Parquet 文件
- 连接外部数据库

### 亮点 5: Pyodide WASM 浏览器运行

```json
// 前端依赖
"pyodide": "0.27.7"
```

marimo 支持在浏览器中完整运行 Python：

- 无需服务器 — 纯客户端执行
- 使用 Pyodide (Python WebAssembly 运行时)
- 支持部分 Python 库 (NumPy, Pandas 等)
- Islands 隔离运行模式
